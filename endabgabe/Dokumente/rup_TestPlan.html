Test Plan

1.     Introduction

1.1     Purpose

Der Zweck des Iteration Test Plan ist es, alle notwendigen Informationen zur Planung und Steuerung des Testaufwands für eine bestimmte Iteration zu sammeln. Es beschreibt den Ansatz für die Prüfung der Software und ist der Top-Level-Plan generiert und genutzt von Managern, den Testaufwand zu lenken. Dieser Testplan für BPM unterstützt die folgenden Ziele

  • Automatisierte Tests (Calabash)
  • Manuelle Anwendungs-Test
  • Test-Vorgehensweise
  • Test-Ressourcen und Aufwände

1.2     Scope

  • Functional Testing (JUnit, Backend)
  • Userinterface-Testing (Calabash)

1.3     Intended Audience

  • Project Manager
  • Project Members

1.4     References

2.     Evaluation Mission and Test Motivation

2.1     Background

Ziel und Motivation hinter den Tests ist die korrekte Funktionsweise der Applikation zu gewährleisen.

2.2     Evaluation Mission

- Auffinden von Fehlern in Logik und Ablauf
- Sicherstellen einer ausreichenden Performance für die Bedienung
- Überprüfen der Erfüllung der Spezifikation

2.3     Test Motivators

- Use Cases
- Performance
- Ablauf

4.2     Testing Techniques and Types

4.2.1     Functional Testing

Technique Objective:

Sicherstellen der Funktionsweise der zugrunde liegenden Algorithmen.

Technique:

Basierend auf JUnit Tests werden Daten an die Algorithmen gesendet und die spezifizierten Antworten zurück erwartet.

Oracles:

Ergebnis der JUnit-Assertions und das Test-Log

Required Tools:

JUnit (in Eclipse integriert)

Success Criteria:

Alle Tests für einen spezifizierten Zustandsübergang müssen erfolgreich sein (grün gehen).

Special Considerations:

Die Tests müssen Anfragen und Antworten gemäß der Kommunikations- Spezifikation versenden bzw. erwarten.


4.2.1     Userinterface-Testing

Technique Objective:

Funktionalität der grafischen Nutzeroberfläche gewährleisten
Technique:

Nutzeraktionen (Eingaben auf dem Touchscreen) werden durch ein Tool emuliert und die danach vorherrschende Oberfläche mit einem definierten Soll-Zustand verglichen.

Oracles:

Der Erfolg des Tests ist gegeben, wenn die grafische Oberfläche einen bestimmten Zustand eingenommen hat. Der Eintritt dieses Zustands kann festgestellt werden, indem man automatisiert auf die Existenz von GUI- Komponenten (Schaltflächen, Eingabefelder, Bilder, ...) prüft.

Required Tools:

Calabash-Android

Success Criteria:

Alle Oberflächen-Tests müssen erfolgreich sein (grün gehen).

Special Considerations:

keine
